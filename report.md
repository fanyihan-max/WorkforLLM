# Gemma-3-270M LoRA 微调报告

## 1. 实验尝试

### 1.1 环境与工具
- **平台**：Google Colab（免费 GPU）
- **基座模型**：`google/gemma-3-270m-it`（2.7 亿参数）
- **微调方法**：LoRA（Low-Rank Adaptation）
  - LoRA rank: 8
  - LoRA alpha: 32
  - 目标模块：`q_proj`, `v_proj`
- **训练超参数**：
  - 学习率：`1e-4`
  - 训练轮数：2
  - Batch size：2（梯度累积 8 步，等效 batch size 16）
  - 精度：FP16（环境不支持 BF16）
- **训练数据**：`logo-detailed-prompt` 数据集（219 条训练样本）
- **优化目标**：自定义程序化奖励函数（`reward.py`），检查 SVG 有效性、标签闭合、坐标合理性、关键词覆盖和防退化

### 1.2 遇到的挑战与调整
- **AI Studio 环境不兼容**：原定使用的百度 AI Studio 环境基于 PaddlePaddle，无法安装 PyTorch，因此迁移至 Google Colab。
- **模型下载被墙**：Hugging Face 的 Gemma 模型为 gated repo，且国内访问受限，改用 ModelScope 的 git 仓库克隆。
- **依赖版本冲突**：初始 `torchao` 版本（0.10.0）与 PEFT 不兼容，卸载后解决。
- **训练参数报错**：
  - `evaluation_strategy` 在新版 Transformers 中已弃用，改为 `eval_strategy`。
  - `Trainer` 不再接受 `tokenizer` 参数，已移除。
  - `bf16` 不被环境支持，改为 `fp16`。
  - `DataCollatorForSeq2Seq` 不适用于 decoder-only 模型，改用固定长度填充 + 默认 collator。
- **训练成功**：经过多次调试，训练顺利完成，loss 从 3.61 下降至 3.085。

## 2. 基座 vs 微调：量化结果

| 模型                     | 平均奖励分数（代理指标） |
| ------------------------ | ------------------------ |
| 基座模型（Gemma-3-270M） | 0.5000                   |
| LoRA 微调模型            | 0.5000                   |
| **提升**                 | **0.0000**               |

**结果解读**：代理指标上完全没有提升。训练 loss 下降表明模型确实学到了训练数据的统计规律，但奖励函数未能捕捉到这些改进。

---

## 3. 为什么没有提升？——深度分析

### 3.1 奖励函数过于宽松，区分度不足
当前 `reward.py` 的评分逻辑：
- 只要输出包含 `<svg>` 标签，基础分即为 **0.2**
- 标签闭合检查几乎所有有效 SVG 都能通过（+0.1）
- 坐标检查假设 viewBox 为 `0 0 100 100`，但模型生成的 SVG 可能使用其他尺寸，导致该检查经常跳过（+0 或 +0.1）
- 关键词覆盖要求较长词（>3 字符），但生成内容往往很短，匹配率低（+0~0.1）
- 长度防退化检查（+0.1 或 -0.2）

综合下来，**几乎所有有效的 SVG 输出都会稳定在 0.5 左右**，即使模型提升了语义匹配度或图形复杂度，分数也无法体现。这导致基座和微调模型的得分完全一样。

### 3.2 模型能力限制
- Gemma-3-270M 只有 2.7 亿参数，远小于 Sonnet 等大模型（几百亿参数）。
- 训练数据仅 219 条，对于生成式任务来说非常有限。
- 模型生成的 SVG 大多非常简短（如 `<svg viewBox="0 0 100 100"><circle cx="50" cy="50" r="20"/></svg>`），结构简单，无法触发奖励函数中更细致的加分项。

### 3.3 训练与推理格式对齐问题
训练时使用 `apply_chat_template` 生成完整对话格式：
```
<start_of_turn>user\n{prompt}<end_of_turn>\n<start_of_turn>model\n{svg}<end_of_turn>
```
推理时使用 `add_generation_prompt=True` 生成输入：
```
<start_of_turn>user\n{prompt}<end_of_turn>\n<start_of_turn>model\n
```
格式是对齐的，但模型在推理时可能仍然输出包含 `user` 部分的重复内容（如诊断结果所示），导致提取的 SVG 内容不准确，进一步影响了奖励分数。

## 5. 对 Goodhart 效应的反思

> *“当一项指标成为目标时，它就不再是一个好指标。”*

在本实验中，代理奖励函数（`reward.py`）是我们优化的目标，但最终它未能反映真实质量。模型学会了生成“有效但朴素”的 SVG——满足了所有程序化检查，但视觉质量和语义匹配度没有提升。

**这其实是 Goodhart 效应的典型体现**：
- 我们优化的是程序化分数，而不是人类视觉评分。
- 模型在训练过程中倾向于“钻空子”，用最简单的 SVG 结构换取稳定的 0.5 分，而不是追求更复杂、更符合提示词的输出。

这也解释了为什么私有测试集上的真实评分（Sonnet 视觉评审）可能会与我们的代理指标产生偏差——即使代理分数没有提升，真实质量可能有一点点进步，但被奖励函数的粗粒度掩盖了；或者代理分数虚高，但真实质量没有变化。

---

## 6. 结论

- **训练有效**：loss 下降证明模型学到了训练数据的分布。
- **提升未被检测**：奖励函数设计过于宽松，无法区分基座和微调模型的差异。
- **模型能力有限**：270M 参数 + 219 条数据，无法期望生成复杂的 SVG。
- **诚实评价**：虽然代理指标没有提升，但训练过程本身是成功的，模型从“无法生成有效 SVG”进步到“能生成简单但有效的 SVG”。这在资源受限的小模型场景下是合理的结果。
